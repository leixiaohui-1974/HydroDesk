# HydroDesk 离线模式设计

## 目标

HydroDesk 需要在断网或弱网环境下仍能提供核心功能，以满足水利工程现场操作需求。

## 离线能力分级

| 级别 | 描述 | 功能范围 |
|------|------|----------|
| L0 | 完全在线 | 全部功能可用 |
| L1 | 弱网 | 大部分功能可用，MCP调用可能延迟 |
| L2 | 本地服务可用 | Ollama + 本地缓存，无远程引擎 |
| L3 | 完全离线 | 仅预缓存数据和本地计算 |

## 本地 LLM：Ollama + Qwen-2.5-7B

### 选型理由

- **Qwen-2.5-7B**：中文能力强，7B参数适合桌面端 GPU/CPU 推理
- **Ollama**：统一的本地推理接口，跨平台支持
- **推理速度**：RTX 3060+ 可达 30+ tokens/s，CPU 约 5-10 tokens/s

### 功能覆盖

1. **意图路由**：将用户自然语言输入分类为具体操作意图
   - 输入："查一下管道水力计算公式" -> `query_knowledge`
   - 输入："运行一下稳态分析" -> `run_simulation`
   - 输入："安阳段怎么了" -> `check_status`

2. **知识问答**：基于缓存文档的 RAG 问答
   - 检索本地 Chroma 向量库
   - 将相关段落作为 context 传给 LLM
   - 生成带引用的回答

3. **文本生成**：报告摘要、操作建议等

### 配置

```
Ollama 地址：http://localhost:11434
默认模型：qwen2.5:7b
备选模型：qwen2.5:14b（需 16GB+ 显存）
           deepseek-r1:7b（推理能力强）
```

## 本地向量数据库：Chroma

### 用途

- 存储预下载的知识文档向量
- 支持语义搜索（离线知识检索）
- 数据持久化到本地文件

### 预缓存策略

```
优先缓存：
├── 运行管理规程
├── 应急处置方案（所有等级）
├── 水力学计算手册
├── EPANET 用户手册
└── 常用设计规范
```

### 存储预算

| 数据类型 | 预估大小 |
|----------|----------|
| 知识文档向量 | 200-500 MB |
| 文档原文缓存 | 100-300 MB |
| 模型文件缓存 | 50-200 MB |
| Ollama 模型 | 4-8 GB（独立管理） |

## 离线预设：应急响应 SOP

预下载的标准操作程序（SOP），在完全离线时可直接查阅：

1. **突发水污染应急处置**
   - 分级响应标准
   - 处置流程图
   - 应急资源清单
   - 联系人列表

2. **管道爆管应急处置**
   - 隔离阀操作指南
   - 水量损失评估
   - 临时供水方案

3. **闸门故障应急处置**
   - 手动操作流程
   - 上下游水位控制
   - 备用通道切换

4. **暴雨洪水应急调度**
   - 预警等级标准
   - 调度预案切换
   - 泄洪闸操作规程

## 优雅降级策略

```
检测连接状态
├── HydroMind 可达？
│   ├── 是 → 全功能模式 (L0)
│   └── 否 → Ollama 可用？
│       ├── 是 → 本地 AI 模式 (L2)
│       │   ├── 意图识别 → 本地 LLM
│       │   ├── 知识检索 → 本地 Chroma
│       │   ├── 仿真计算 → 本地 EPANET
│       │   └── MCP 调用 → 不可用，提示用户
│       └── 否 → 纯离线模式 (L3)
│           ├── 预缓存文档 → 可查阅
│           ├── 应急 SOP → 可查阅
│           ├── 模型编辑 → 可操作
│           └── 计算/AI → 不可用
```

## 数据同步机制

### 离线变更队列

所有离线期间的操作自动记录到同步队列：

```javascript
{
  type: 'model_update',     // 操作类型
  data: { ... },            // 操作数据
  timestamp: 1710000000000, // 时间戳
  id: 'sync-xxx'            // 唯一 ID
}
```

### 恢复连接后同步

1. 自动检测连接恢复
2. 按时间顺序回放变更队列
3. 冲突检测与解决（默认策略：最后写入胜出）
4. 同步完成后清空队列
5. 拉取远端最新数据

### 冲突处理

- **模型冲突**：保留两个版本，提示用户合并
- **配置冲突**：本地优先
- **数据冲突**：时间戳最新优先
